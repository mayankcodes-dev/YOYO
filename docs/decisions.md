## ADR-001: Remove Clerk, Replace with Custom JWT Auth

**Date:** 2026-06-28

**Decision:** Removed `@clerk/clerk-react` and `@clerk/express`. Replaced with custom JWT authentication (jsonwebtoken + bcryptjs) and Google OAuth via `google-auth-library` + `@react-oauth/google`.

**Reason:** Portfolio project — Clerk adds vendor lock-in, cost, and unnecessary complexity for a showcase. Custom JWT demonstrates deeper auth knowledge to recruiters.

**Alternatives Considered:**
- Keep Clerk — rejected (vendor dependency, not educational)
- NextAuth — rejected (project uses separate Express backend, not Next.js)

**Tradeoffs:**
- ✅ More control over auth flow, demonstrates skills
- ✅ No external auth service dependency
- ❌ More code to maintain (token refresh, session management)
- ❌ Manual user sync between auth and DB required

**Impact:**
- User._id is now MongoDB ObjectId (not Clerk's `user_xxx` string)
- Backend middleware rewrote from `clerkMiddleware` to custom `protect` (JWT verify)
- Frontend token stored in localStorage as `yoyo_token`
- Legacy Clerk user IDs in DB handled with `strictPopulate: false` + filter in `getRooms`

## ADR-002: Remove Razorpay from Active UI, Keep Stripe as Primary

**Date:** 2026-06-28

**Decision:** Razorpay integration code is kept in `bookingController.js` but is NOT exposed in the UI. Stripe is the only active payment gateway shown to users.

**Reason:** Razorpay requires business KYC verification for live payments. As a portfolio project with no real transactions, this is a blocker. Stripe test mode works without KYC.

**Alternatives Considered:**
- Remove Razorpay entirely — rejected (code demonstrates broader knowledge)
- Use Razorpay in test mode — rejected (UX is worse than Stripe; Stripe test checkout is cleaner)

**Tradeoffs:**
- ✅ Stripe works fully in test mode
- ✅ Keeping Razorpay code shows awareness of Indian payment ecosystem
- ❌ Razorpay code is "dead code" unless activated

**Impact:** `VITE_RAZORPAY_KEY_ID` in client env is not used by any active UI component.

## ADR-003: Hero Section — Cloudinary Video Instead of Static Image

**Date:** 2026-06-29

**Decision:** Hero section background changed from a local static PNG to a Cloudinary-hosted MP4 video.

**Reason:** User requested a more visually impactful hero. Video creates a premium feel. Cloudinary handles transcoding and CDN delivery.

**Alternatives Considered:**
- Keep static PNG — rejected (less visual impact)
- Self-host video in `public/` — rejected (large file in git repo, slow loads)

**Tradeoffs:**
- ✅ Premium visual impact
- ✅ Cloudinary handles CDN + adaptive streaming
- ❌ Depends on Cloudinary availability (static fallback not implemented)

**Impact:** `Hero.jsx` uses Cloudinary URL `v1782728427/hero_video_main_teqgjw.mp4`. Fallback is a `<poster>` static image.

## ADR-004: Testimonial Carousel — AnimatePresence popLayout (All 3 Cards Shift)

**Date:** 2026-06-29

**Decision:** Testimonial carousel uses Framer Motion's `AnimatePresence mode="popLayout"` with a flex row of 3 cards. All 3 cards shift their physical position when advancing. The entering card slides in from one side; the exiting card slides out the other.

**Reason:** User requested "the remaining two testimonials should shift toward the direction of scroll, and bring a new testimonial from the opposite side." `popLayout` achieves this by animating the layout shift of non-exiting elements.

**Alternatives Considered:**
- Single card fade only (previous implementation) — rejected (felt static)
- CSS transform translateX with overflow:hidden — rejected (harder to synchronize with React state)
- Grid-based approach with only the new column animating — rejected (previous version, didn't give the conveyor belt feel)

**Tradeoffs:**
- ✅ True carousel "belt" feel
- ✅ Uses stable Framer Motion API
- ❌ Requires testimonials to have a stable `id` key per object (AnimatePresence needs unique keys)

**Impact:** `Testimonial.jsx` relies on `t.id` as the AnimatePresence key. The `testimonials` array in `assets.js` must have unique `id` on each object or animations will mismatch.

## ADR-005: User._id Type — MongoDB ObjectId (Post-Clerk Migration)

**Date:** 2026-06-28

**Decision:** After removing Clerk, User._id is now a standard MongoDB ObjectId, auto-generated by Mongoose. All refs (Hotel.owner, Booking.user) are ObjectId.

**Reason:** Clerk's User IDs were strings (`user_xxx`). The original schema used String for all User refs to match. Now that auth is custom, ObjectId is the correct type.

**Historical Note:** Some documents in the DB from the Clerk era may still have string `user_xxx` IDs. The `getRooms` controller uses `strictPopulate: false` and filters out these orphaned records gracefully.

**Impact:**
- `authMiddleware.js` extracts `req.user._id` which is an ObjectId
- Any code doing `userId === someStringId` comparisons will break — always use `.toString()` for comparison if needed

## ADR-006: Email Provider — Gmail SMTP via Nodemailer

**Date:** Inferred from codebase (pre-session)

**Decision:** Email notifications sent via Gmail SMTP using Nodemailer. Credentials stored in `NODEMAILER_USER` and `NODEMAILER_PASS` (Gmail app password).

**Reason:** Free for portfolio-scale usage. No third-party email service cost.

**Alternatives Considered:**
- Brevo (formerly Sendinblue) — was used previously (per `.ai-context.md`), replaced with Gmail
- SendGrid — rejected (requires domain verification for production)

**Tradeoffs:**
- ✅ Free, zero config beyond Gmail app password
- ❌ Gmail rate limits (500 emails/day)
- ❌ Email sending is synchronous in the request path (blocks response)

**Impact:** `configs/nodemailer.js` uses `smtp.gmail.com:587`. Future: move email to a background job (fire-and-forget).

## ADR-007: Real-time Features — Socket.io

**Date:** Inferred from codebase

**Decision:** Socket.io added for real-time features (live viewer count on room listings).

**Reason:** Demonstrates WebSocket capability on the portfolio. LiveViewers component connects to the socket server on mount.

**Alternatives Considered:**
- Server-Sent Events — rejected (socket.io already widely known; more impressive on resume)
- Long polling — rejected (inefficient)

**Tradeoffs:**
- ✅ Bidirectional real-time capability
- ✅ `socket/` directory exists for future expansion
- ❌ Adds complexity to Vercel serverless deployment (requires persistent connection; may need a separate server)

**Impact:** `server.js` creates an `http.Server` wrapping the Express app, then passes to `socket.io`. `server.listen()` is used instead of `app.listen()`.

## ADR-008: AI Chatbot — Groq SDK (Maya)

**Date:** Inferred from codebase

**Decision:** Maya AI chatbot uses Groq's LLM API via the `groq-sdk` npm package.

**Reason:** Groq provides free tier with very fast inference. Demonstrates AI integration skills.

**Alternatives Considered:**
- OpenAI API — rejected (paid, rate limits stricter on free tier)
- Anthropic Claude API — rejected (cost)
- Local model (ollama) — rejected (requires GPU server)

**Tradeoffs:**
- ✅ Fast responses (Groq specializes in low-latency inference)
- ✅ Free tier sufficient for portfolio
- ❌ Groq API key required; if quota exceeded, chatbot fails

**Impact:** `aiController.js` uses Groq SDK. Support email in system prompt: `admin@mayankcodes.dev`. Fallback message displayed if API fails.

## ADR-009: Image Storage — Cloudinary (Cloud: dgqgzmzed)

**Date:** Inferred from codebase

**Decision:** All user-uploaded images (room photos, hotel photos, avatars) stored on Cloudinary. A `optimiseImage(url, width)` utility in `client/src/utils/cloudinary.js` automatically applies `f_auto,q_auto,w_N` transformations.

**Reason:** CDN delivery, automatic format conversion (WebP), quality optimization without manual effort.

**Alternatives Considered:**
- Local disk storage — rejected (doesn't persist on serverless, no CDN)
- AWS S3 — rejected (cost, more complex setup for portfolio)

**Tradeoffs:**
- ✅ CDN, automatic WebP, free tier
- ✅ `optimiseImage()` helper makes usage consistent
- ❌ Cloudinary free tier has storage and transform limits

**Impact:** All image URLs in the DB are Cloudinary `secure_url` values. Always pass through `optimiseImage()` for display.

## ADR-010: PWA Support — vite-plugin-pwa

**Date:** Inferred from codebase

**Decision:** Added `vite-plugin-pwa` to enable Progressive Web App capabilities (service worker, install prompt, offline support).

**Reason:** Demonstrates modern web app capabilities. Improves Lighthouse score. Shows mobile-readiness.

**Tradeoffs:**
- ✅ Installable, offline-capable, improved performance scores
- ❌ Service worker cache can cause stale data issues in dev — always hard-refresh when testing

**Impact:** `PWAInstallBanner.jsx` and `PWAUpdatePrompt.jsx` components handle install and update UX. `vite.config.js` includes PWA plugin config.
